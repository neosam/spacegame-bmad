## Successes and Strengths

### B0002 Buffer Pattern — Established and Applied Consistently
The Bevy 0.18 MessageReader+MessageWriter conflict (B0002) requires separate systems for reading and writing to the same Messages resource. This was encountered in Stories 3.3, 3.4, and 3.7, and each time the buffer pattern was applied correctly: `PendingCreditEvents`, `PendingDropSpawns`, `PendingPickupEvents`. Pattern is now an established team convention.

### Pure Function Design for Testability
`decide_material_drop(entity_type, roll)` in 3.4 set the template: pure function, takes an explicit random roll, no side effects, fully testable without App setup. In Story 3.7, adding a `tier: u32` parameter was trivial exactly because the function was already designed for testability. This pattern should be the default for any probabilistic or scaling logic.

### Circular Import Prevention (MaterialType Placement)
Placing `MaterialType` in `src/shared/components.rs` rather than `economy.rs` or `events.rs` proactively prevented a circular import chain that would have required significant restructuring. Recognized early in Story 3.4.

### Save System Migrations: Zero Data Loss
Two SAVE_VERSION bumps (2→3 and 3→4) with `#[serde(default)]` on new fields. Old saves continue to deserialize correctly with default values. No save corruption across the entire epic.

### Bevy 0.18 UI Patterns Solidified
`Text` as standalone component (not `TextBundle`), `GlobalZIndex` for layering, `Added<Docked>` / `RemovedComponents<Docked>` for event-driven UI. These patterns are now confirmed and documented by usage.

## Challenges and Growth Areas

### DiscoveredChunks vs ExploredChunks Conceptual Confusion
Two resources serve near-identical purposes: `DiscoveredChunks` (prevents double-crediting) vs `ExploredChunks` (map tracking). The distinction is correct architecturally but creates cognitive overhead. Future developers reading economy.rs alongside world/mod.rs may conflate them.

### Test App Setup Incomplete at Story Creation Time (earn_credits.rs)
Story 3.3 created `credits_test_app()` without `WorldConfig`. This was fine at the time (award_credits_on_kill didn't need it). Story 3.7 added `Res<WorldConfig>` to that system — and `credits_test_app()` needed a fix that wasn't in the File List. Detected in code review but represents a gap: test apps should be set up with a "full production resource set" from the start.

### Lingering Warnings in Test Files (Pre-existing, not Epic 3 caused)
Tests in `event_system.rs`, `tutorial_zone.rs`, `station_docking.rs` have unused variable / unused import warnings. Not introduced by Epic 3, but none of the Epic 3 code reviews addressed them. The code review scope was limited to the story's File List rather than the wider warning surface.

### mod helpers; Stale Include
`tests/earn_credits.rs` had `mod helpers;` included since Story 3.3 but never used helpers. This produced 2 warnings for 4 subsequent stories before being cleaned up in the Story 3.7 code review. Indicates code review should scan for pre-existing warnings in all touched files, not just new code.

## Key Insights and Learnings

1. **Buffer pattern is the standard for multi-system event flows** — Any system that reads AND writes the same `Messages<T>` resource needs a pending buffer resource. This pattern is battle-tested and should be part of the new-system checklist.

2. **Pure functions + explicit random inputs = frictionless scaling** — `decide_material_drop(entity_type, roll, tier)` was extended from 2 to 3 args in Story 3.7 with zero test breakage (only `tier: 0` additions). Design for extensibility by keeping business logic in pure functions.

3. **Test app setup should mirror production resource initialization** — If a system will ever plausibly need a resource, include it from the start. `WorldConfig::default()` is cheap and prevents future breaking changes.

4. **Code review scope = File List + all files those files touch** — A warning in `earn_credits.rs` survived 4 story reviews because reviewers only checked "changed files". The next-level rule: check warnings in any test file that imports code from the story's File List.

5. **`#[serde(default)]` for every new save field** — Never add a new field to PlayerSave without `#[serde(default)]`. This is now a hard convention.

## Successes and Strengths

### Core/Rendering Marker Pattern
The `NeedsXVisual` marker pattern (core spawns entity with marker, rendering attaches mesh/material, removes marker) was applied consistently across all 8 stories and scaled well. This clean architectural boundary kept core systems testable without rendering infrastructure.

### RON Config Pattern
The `from_ron()` + `Default` fallback pattern was used three times (FlightConfig, WeaponConfig, SpawningConfig) with zero issues. Reliable, simple, consistent.

### Test Discipline
124 tests (65 unit + 59 integration) with zero regressions across all stories. The `test_app()` harness evolved organically and remained maintainable. Every acceptance criterion has at least one test.

### Code Review Value
Every story was code-reviewed. Reviews found real bugs: system ordering in Story 0.6, GPU asset leak, missing tests, stale comments. The adversarial review process caught issues that testing alone missed.

### DamageQueue Buffer Pattern
Elegant solution for cross-SystemSet communication (Collision → Damage). Buffered damage entries allow multiple hits per frame and clean separation of detection from application.

### Bevy 0.18 Adaptation
Events → Messages was the biggest API surprise (Story 0.2). After the initial learning curve, the team adapted quickly and subsequent stories were smooth.

## Challenges and Growth Areas

### Cross-Schedule Communication Complexity
Story 0.6 (Visual Feedback) was the most complex — 2 code review rounds, system ordering bug, GPU asset leak. FixedUpdate → Update resource bridging (DestroyedPositions, LaserHitPositions) is a complexity hotspot requiring careful ordering.

### Laser Visual Disconnect (Bug)
LaserPulse spawned as independent entity with fixed Transform. When the player moves or rotates, the laser flash stays at the old position. Discovered through player feedback, not testing or code review. **Fix planned:** Make LaserPulse a child entity of the player ship.

### Late Balance Discovery
Ship speed felt too slow and drag too high — only discovered when Simon actually played. No automated test can catch "feels sluggish." Speed values in `flight.ron` were tuned to satisfaction, but the discovery was late.

### No Audio Integration
`bevy_kira_audio` has been a dependency since Story 0.1 but was never wired up across 8 stories. Audio is critical for game feel but was consistently deferred.

### No Hot-Reload for Config
FlightConfig loaded via `std::fs::read_to_string` at startup — no hot-reload. Makes iterative balance tuning painful (restart required for every change). Inspector-based live tuning or AssetServer migration needed.

---

## Key Insights and Learnings

1. **Player feedback beats code review for game feel** — Balance issues (speed, drag) and visual bugs (laser disconnect) were only found through actual play. Automated tests validate logic, not feel.

2. **Child entity pattern for attached visuals** — Simon's suggestion to make LaserPulse a child of the player ship, leveraging Bevy's transform propagation instead of manual tracking. More idiomatic, less code, better result.

3. **Config values must be live-tweakable** — For tuning sessions, restarting the game per change is unacceptable. Inspector exposure or AssetServer hot-reload is essential.

4. **Marker pattern scales** — The NeedsXVisual pattern worked for lasers, projectiles, asteroids, and drones without modification. It's a proven pattern for core/rendering separation.

5. **Cross-schedule communication needs care** — FixedUpdate → Update data passing via resources works but requires explicit clear-after-consume and careful system ordering.

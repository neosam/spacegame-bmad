## Successes and Strengths

### State Machine Architecture
7-phase TutorialPhase state machine (Flying → Shooting → SpreadUnlocked → Complete → StationVisited → GeneratorDestroyed → TutorialComplete) implemented cleanly. All systems use phase guards, all transitions are idempotent. No race conditions.

### 100-Seed Fuzz Testing
Automated validation across 100 seeds catches layout constraint violations that manual testing never would. This pattern scales and should be reused for any procedurally-placed content.

### Established Patterns
- `JustDamaged` as per-frame damage signal — clean, no polling
- `OnEnter`/`OnExit` schedules for phase-dependent system activation
- `CascadeTimer` resource for time-delayed actions
- Idempotency guards on all phase transitions
- `warn!()` startup validation (following validate_speed_cap pattern)

### YOLO Sprint Workflow
8 stories implemented without per-story manual review. Sprint-end playtest is the right moment for Simon's input. Confirmed working.

### Camera Bug Found and Fixed Immediately
Playtest found the duplicate Player spawn bug within minutes. Root cause: `spawn_tutorial_zone` spawned a second Player entity (no mesh), causing `camera_follow_player`'s `.single()` to fail silently. Fix: removed player spawn from tutorial zone, `setup_player` in rendering now reads `TutorialConfig + WorldConfig` to position player at tutorial layout spawn.

## Challenges and Growth Areas

### Invisible Tutorial Entities (Critical Gap)
TutorialStation, TutorialWreck, and GravityWellGenerator were spawned without any rendering components. All are invisible in-game. Each story dev noted "No rendering in this story" and deferred it — but no story ever picked it up.

### Missing Flying → Shooting Trigger (Critical Gap)
The `Flying → Shooting` phase transition (player approaches wreck → gets laser) was never implemented. `advance_phase_on_wreck_shot` transitions `Shooting → SpreadUnlocked`, but the first transition has no trigger system. The tutorial progression cannot start.

### No Gravity Well Visual Feedback (Medium Gap)
The gravity well pull is functional but invisible. Players feel an invisible force with no visual boundary indicator. Disorienting.

### Definition of Done Did Not Include Playability
"All checkboxes green" + "tests pass" was treated as done. A story can have passing tests and be mechanically correct but completely non-functional from a player perspective. The Rendering gap was not caught because no story had "player can see this entity" as an explicit acceptance criterion.

### Duplicate Player Spawn Pattern
`spawn_tutorial_zone` duplicated the Player spawn that rendering already handled. The core/rendering architectural separation (core spawns entity with marker, rendering adds visual) was violated. This caused the camera to freeze.

---

## Key Insights and Learnings

1. **Playtest catches what 430 tests cannot** — Simon's 2-minute playtest found 4 issues (camera bug + 3 gameplay gaps). Tests verify mechanics; playtest verifies experience.

2. **"Done" must include "player can see and interact with this"** — Any entity that the player must perceive or interact with needs a rendering component before the story is done. No invisible gameplay entities.

3. **Core/Rendering separation must be respected** — Core spawns entity + marker components. Rendering adds visuals. Violating this (tutorial spawning a full Player) causes subtle bugs that are hard to diagnose.

4. **State machine completeness needs end-to-end verification** — Individual phase transitions can all be tested in isolation while a gap between phases goes undetected. Need at least one integration test that walks the full happy path.

5. **Visual feedback is not polish — it's functionality** — The gravity well boundary, the station appearance, the wreck location: these are not art. Without them, the player has no information to act on.

Four separate systems used the pure function design:
- `next_state(&AiState, &AiContext) -> AiState` — AI FSM (established in epic4-prep)

## Challenges and Growth Areas

### DoD Violation: New Enemy Types Were Invisible
**What happened:** Stories 4-2 (Fighter), 4-3 (HeavyCruiser), 4-4 (Sniper), and 4-10 (TraderShip) were implemented with `NeedsXxxVisual` markers but no rendering systems were added to `rendering/mod.rs`. All four enemy types were invisible in the running game.

**How found:** Simon ran the game manually and noticed only red Scout Drones. "Es fliegen ein paar rote Dinge rum" — the new enemies were completely invisible.

**Fix:** Mini-sprint 4-13 added `generate_fighter_mesh`, `generate_heavy_cruiser_mesh`, `generate_sniper_mesh`, `generate_trader_mesh` to `vector_art.rs` plus four Asset resources and render systems in `mod.rs`. +4 tests. Fixed in the same session.

**Root cause:** Story batching (4-2+4-3+4-4 in one commit) combined with no live gameplay verification. `cargo test` passed — but tests do not verify that entities are visible on screen.

### DoD Violation: Player Health + Energy HUD Never Implemented
**What happened:** The GDD and Epic 0 scope explicitly included "Basic HUD (health, energy)". This was never implemented as a story. Four epics passed without the player ever seeing their own health bar or energy level.

**How found:** Simon asked "Wann wird eigentlich meine eigene Health angezeigt?"

**Impact:** Player has no feedback for:
- How close to death they are
- Energy available for Spread weapon
- Whether Hull Strength upgrade (Epic 5) has any effect

**Fix:** Mini-sprint 4-14 (planned) before Epic 5 starts.

### No Story Documentation Files Created
The automated dev agent implemented all 12 stories directly without creating `4-X-story-name.md` files. All previous epics (1-3) had complete story documentation. This means:
- No dev notes or implementation log for Epic 4
- No per-story review feedback captured
- No explicit DoD checklist per story

### Story Batching Hides Granularity
Commits like `feat(4-2+4-3+4-4)` and `feat(4-7+4-9)` mean 5 stories have combined commit history. This made it harder to identify which specific story caused the rendering gap.

## Key Insights and Learnings

1. **`cargo test` ≠ DoD for visual entities** — A Bevy entity can have 0 failing tests and be completely invisible at runtime. The Sichtbarkeits rule requires manual gameplay verification, not just test execution.

2. **Story batching is a visibility risk** — When multiple entity types are implemented in one commit, the rendering check is easily missed. Max 2 stories per commit, each requiring explicit visual confirmation.

3. **epic4-prep pattern is high-value** — Establishing shared types, pure functions, and skeleton plugin before a sprint consistently accelerates Story 1. This should be standard for every epic with new subsystems.

4. **Scope items from earlier epics can survive many sprints undetected** — The Player HUD was Epic 0 scope. It survived Epic 1, 2, 3, and 4 without anyone noticing. A "feature completeness check" against the GDD should be part of each epic retrospective.

5. **Story files are not optional process overhead** — They are the primary mechanism for capturing gotchas, dev notes, and review feedback. Without them, the team has no institutional memory of why decisions were made.
